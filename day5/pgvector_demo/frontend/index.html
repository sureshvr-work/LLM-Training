bring vectors to your existing database
CREATE EXTENSION IF NOT EXISTS vector;

Loads pgvector into this database. Postgres is a small core plus a plug-in system — an extension adds new types, operators, functions and index methods into one database, without changing Postgres itself. This one line is the whole “install the vector DB” step, and it gives you three things at once:

+ a column typevector(n) · halfvec · bit · sparsevec — a column can hold a fixed-length array of floats.
+ operators<=> cosine · <-> L2 · <#> inner product · <+> L1 — for comparing vectors.
+ index typesHNSW & IVFFlat — the approximate-nearest-neighbour indexes that make similarity search fast.
Per-database. Run it in each database that needs vectors — not once per server.
The extension files must already be on the machine (a package, or baked into an image like pgvector/pgvector:pg16). CREATE EXTENSION activates what's installed — it doesn't download anything.

④ Normal data vs vector data — the differences

Normal data

  • Value is the data — small scalar (int, text).
  • Stored as-is; no model involved.
  • Index: B-tree — exact lookup & ranges.
  • Query: WHERE col = … — equals / LIKE.
  • Exact & deterministic — matches or it doesn't.

Vector data

  • Fixed-length float array from an embedding model.
  • Extra step on write & read: text → vector.
  • Index: HNSW / IVFFlat — approximate NN.
  • Query: ORDER BY embedding <=> $q — nearest.
  • Approximate & ranked — finds by meaning; recall is tunable.

The invariant: the query must be embedded with the same model as the rows — same idea as the analyzer rule in the keyword pipeline, different machine.
