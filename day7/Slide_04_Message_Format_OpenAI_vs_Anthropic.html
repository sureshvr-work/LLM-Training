The prompt is everything

The agent harness does one thing: build a request, send it, read the reply, append it, repeat. That request — system + messages + tools — is the whole prompt. Here's exactly what it looks like on the wire. Flip between providers; click a part to open it.

Same three inputs as the anatomy slide — system (the Goal), messages (the Memory), tools (the catalog) — now as literal JSON. The model is a pure function of this payload; nothing else exists between turns.
ANTHROPIC Anthropic · Messages API
system — the Goal · role + objective ›
Where the standing instruction lives. This is the one structural fork between the two APIs.
# Anthropic — `system` is its own TOP-LEVEL field, beside messages
{
  "model": "claude-...",
  "max_tokens": 1024,
  "system": "You plan trips and book within budget.",   ← top-level
  "messages": [ /* the conversation */ ],
  "tools":    [ /* the catalog   */ ]
}
# OpenAI — `system` is the FIRST message inside the messages array
{
  "model": "gpt-...",
  "messages": [
    { "role": "system",                          ← first message
      "content": "You plan trips and book within budget." },
    /* ...then user / assistant / tool messages... */
  ],
  "tools": [ /* the catalog */ ]
}
⚑Anthropic:OpenAI: the goal is a separate system parameter — it isn't part of the conversation. the goal is just the first role: "system" message — there is no separate field.
messages — the Memory · the whole conversation, re-sent every turn ›
The running history. One slice: the request, the assistant's tool call, and the tool's result coming back. Three real differences hide in here.
# Anthropic — tool call = a content block; tool result rides in a USER message
"messages": [
  { "role": "user",
    "content": "Plan a trip: Goa, 5★, 5 nights, under $5,000" },

  { "role": "assistant",
    "content": [
      { "type": "text", "text": "Searching flights first." },
      { "type": "tool_use", "id": "toolu_01",
        "name": "search_flights",
        "input": { "from": "GOI", "when": "next month" } } ] },  ← input = OBJECT

  { "role": "user",                                  ← result rides in a USER msg
    "content": [
      { "type": "tool_result", "tool_use_id": "toolu_01",
        "content": "12 flights · cheapest $1,180" } ] }
]
# OpenAI — tool call = a tool_calls array; tool result = its own role:"tool" msg
"messages": [
  { "role": "user",
    "content": "Plan a trip: Goa, 5★, 5 nights, under $5,000" },

  { "role": "assistant", "content": null,
    "tool_calls": [
      { "id": "call_01", "type": "function",
        "function": {
          "name": "search_flights",
          "arguments": "{\"from\":\"GOI\",\"when\":\"next month\"}" } } ] },  ← args = STRING

  { "role": "tool", "tool_call_id": "call_01",          ← its own "tool" role
    "content": "12 flights · cheapest $1,180" }
]
⚑Three differences to point at: (1) the call is a tool_use content block; (2) input is a real object; (3) the result comes back as a tool_result block inside a user message. (1) the call is a tool_calls array; (2) arguments is a JSON string you must parse; (3) the result is its own message with role: "tool".
tools — the catalog · what it may call + how to pick ›
Each tool is a name, a description, and an argument schema. The description is the model's only basis for picking the tool — it's prompt real estate, not documentation.
# Anthropic — schema key is `input_schema`, tool sits at top level
"tools": [
  { "name": "search_flights",
    "description": "Find flights between two cities for given dates.",
    "input_schema": {                          ← input_schema
      "type": "object",
      "properties": {
        "from": { "type": "string" },
        "when": { "type": "string" } },
      "required": ["from", "when"] } }
]
# OpenAI — wrapped in {type:"function", function:{...}}, schema key is `parameters`
"tools": [
  { "type": "function",                          ← function wrapper
    "function": {
      "name": "search_flights",
      "description": "Find flights between two cities for given dates.",
      "parameters": {                            ← parameters
        "type": "object",
        "properties": {
          "from": { "type": "string" },
          "when": { "type": "string" } },
        "required": ["from", "when"] } } }
]
⚑The JSON-schema body is identical. Only the wrapper differs: input_schema sits directly on the tool. the tool is wrapped in { "type": "function", "function": { … "parameters": … } }.
the response — one decision back · read the stop signal, then branch ›
This is the Reason box's output. You read one field to decide: did it call a tool, or is it done?
# Anthropic — content blocks + stop_reason; read response.content directly
{
  "id": "msg_...",
  "role": "assistant",
  "content": [
    { "type": "tool_use", "id": "toolu_02",
      "name": "find_hotels",
      "input": { "city": "Goa", "stars": 5 } } ],
  "stop_reason": "tool_use",        ← "tool_use" = call a tool · "end_turn" = final answer
  "usage": { "input_tokens": 1843, "output_tokens": 56 }
}
# OpenAI — nested under choices[0].message; finish_reason carries the signal
{
  "id": "chatcmpl-...",
  "choices": [
    { "index": 0,
      "message": {
        "role": "assistant", "content": null,
        "tool_calls": [
          { "id": "call_02", "type": "function",
            "function": { "name": "find_hotels",
              "arguments": "{\"city\":\"Goa\",\"stars\":5}" } } ] },
      "finish_reason": "tool_calls" }   ← "tool_calls" = call a tool · "stop" = final answer
  ],
  "usage": { /* ... */ }
}
⚑Same branch, different field name: read stop_reason — tool_use → run the tool & loop; end_turn → done. The reply is response.content. read choices[0].finish_reason — tool_calls → run the tool & loop; stop → done. The reply is choices[0].message.
# the entire agent, in five lines: req = { system, messages, tools } # ← the prompt = everything resp = POST(req) # one stateless call if stop_signal == "call a tool": # stop_reason / finish_reason messages += [ the call, run(tool) ] # append → grows the Memory repeat # else: messages holds the final answer — done

Schemas trimmed for teaching; full field lists at docs.claude.com (Messages API) and platform.openai.com (Chat Completions). OpenAI also offers the newer Responses API; the Chat Completions shape above is the one most code uses today.
