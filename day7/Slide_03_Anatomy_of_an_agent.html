Anatomy of an agent

Almost any agent you build reduces to the model below. Click Sense · Reason · Act to open up the work happening inside each step.

The loop. An agent is an LLM in a loop: it senses, reasons, acts — then repeats until the goal is met.
SENSE · gather the world into context ›
It gathers, it doesn't think.
# SENSE just gathers the three inputs the model will read. No thinking here.
GOAL  = "Plan a trip — Goa · 5★ · 5 nights · ≤ $5,000"   # → system
MEMORY = history          # → messages: the WHOLE story so far, re-sent every turn
                          #   the request + every prior step + every tool result
                          #   newest entry = result of the last action
                          #   retrieved docs / RAG chunks get appended here too
TOOLS = [search_flights, find_hotels, book_flight, book_hotel]   # → tools (the catalog)

# The model is stateless — it kept nothing from last turn. So we re-send ALL
# of MEMORY, not just the newest line. More turns → longer history → more tokens → more cost.
Sense writes no logic of its own. It hands Reason three inputs: the goal (system), the whole history (messages), and the tool catalog (tools). Tool descriptions ride in tools; tool results and any retrieved chunks ride in messages.
REASON · weigh options, pick one move ›
The only place a decision happens. Goal, Memory and Tools aren't extra inputs sitting beside the context — they are the context, handed to one stateless call.
# Goal, Memory, Tools aren't extra things beside the context — they ARE the
# context, handed to ONE stateless call as three inputs:
GOAL  = "You plan trips and book within budget."             # ← Goal   (system)
TOOLS = [search_flights, find_hotels, book_flight, book_hotel]   # ← Tools  (tools)

# ───────── THIS ONE CALL IS THE REASON BOX ─────────
response = client.messages.create(
    model    = "claude-...",
    system   = GOAL,       # ← the Goal
    messages = MEMORY,     # ← the Memory: the whole history, from Sense
    tools    = TOOLS,      # ← the Tools it may call
)   # reads all three → returns ONE move
# ────────────────────────────────────────────────────

# Branch on the one decision:
if response.stop_reason == "tool_use":
    call = response.content[-1]        # { name: "search_flights", input: {...} }
    # → CALL A TOOL — hand it to Act.
else:
    answer = response.content[0].text  # → FINAL ANSWER — loop ends.
There's no separate memory variable — memory is the history portion of messages that you re-send every turn. The model is stateless; the harness keeps the history.
ACT · run the tool, catch the result ›
Where a decision becomes a real action.
# Reason returned a tool_use. ACT runs it for real.
name = call.name                 # "search_flights"  ← THE CALL
args = call.input                # {"from": "GOI", "when": "next month"}

result = TOOLS[name](**args)     # actually run it: search_flights(from="GOI", when="next month")
                                 # → "12 flights · cheapest $1,180"  ← RESULT

# feed the result back so it becomes the next turn's Sense:
messages = messages + [
    {"role": "assistant", "content": [call]},
    {"role": "tool", "content": result},   # ← appended to history → next SENSE
]
Reason only named the tool; Act executes it — TOOLS[name](**args) — and appends both the call and its result to messages. That growing list is the memory the next turn re-reads.
