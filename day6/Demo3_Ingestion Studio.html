5 How that chunking cuts
Recursive Chunking
Structured data isn't sliced like prose — one record = one chunk (that boundary is the same for both engines).
• For embedding: serialize each record into readable text first — embedding raw {...} retrieves worse than a readable line.
• For keyword index: no serialize needed — map fields to indexed fields (customer, status, amount) so you can filter & boost; punctuation is ignored anyway.
EMBEDDING PATH — ① serialize → ② one chunk per record (shown below)
chunk 1 — record 1
order_id: A-10293, customer: Acme Corp, amount: 1299.5, status: shipped
chunk 2 — record 2
order_id: A-10294, customer: Globex, amount: 842, status: pending
chunk 3 — record 3
order_id: A-10295, customer: Initech, amount: 5400, status: shipped
Note: N records → N chunks — fine at this scale. If a single nested record is too big for one chunk, split it further (recursive).
Recursive on structured data recurses the structure, not text separators: array → record → sub-object → field, descending only into a node that's still too big. Flat records collapse to one-per-chunk (above); a single fat nested record splits inside itself — e.g. order ▸ items ▸ item 3 — keeping the path as a breadcrumb.
Applies to every structured strategy: the cut follows the structure (records / rows / elements), then each piece is serialized to readable text for embedding — and its fields kept for keyword & filter. The strategy only changes where / how deep you cut, never whether you respect the structure.
Code peek — how this cut is computed (Python)
def record_level(records, serialize):            # structured data
    # 1) serialize each record to readable text, 2) one record = one chunk
    return [serialize(r) for r in records]
    # serialize({"order_id":"A-10293","amount":1299.5,"status":"shipped"})
    #   -> "order_id: A-10293, amount: 1299.5, status: shipped"
    # (embedding the readable line retrieves far better than raw {...})
6 Then: embed, index — or both?
This data is text, so both engines apply — most production RAG runs them together (hybrid: BM25 + vector).
available

Semantic — Embedding

stored as vectors · matched by vector similarity (cosine · dot-product · L2) ⓘ
Embeds the chunk text into a dense vector — text embedder (e.g. text-embedding-3, bge, Cohere).
available

Keyword — Indexing

stored as an inverted index · matched by term scoring (BM25, TF-IDF, …) ⓘ
There's text, so it drops straight into an inverted index — exact terms, filters, field boosts. Classic scorers need no model.
